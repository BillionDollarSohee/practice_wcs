# Kata 3 — TCP 소켓 + 디버깅 연습

`EchoServer`(서버)와 `EchoClient`(클라이언트) 두 프로젝트로 구성됩니다.
프로토콜은 실무 Vision 소켓과 똑같은 **STX(0x02) + 내용 + ETX(0x03)** 프레이밍입니다.

## 실행 순서

1. 터미널 두 개 열기
2. 하나에서: `dotnet run --project 03-tcp-echo/EchoServer`
3. 다른 하나에서: `dotnet run --project 03-tcp-echo/EchoClient`
4. 서버를 먼저 켜야 클라이언트가 접속됩니다.

## 완성 순서 추천

1. `EchoServer`의 TODO 1~9 채우기 (서버 혼자서는 접속만 받고 아무것도 못 하는 게 정상입니다)
2. `EchoClient`의 TODO 1~5 채우기
3. 서버 켜두고 클라이언트 실행 → `"hello"` → `"HELLO"` 로 돌아오는지 확인
4. 5000자짜리 메시지가 안 깨지고 돌아오는지 확인 (여기서 막히면 그게 정상입니다 — TCP 스트림의 핵심을 마주친 거예요)
5. `"QUIT"` 보내면 양쪽 다 깔끔하게 종료되는지 확인

## 디버깅 연습 — 이번 kata의 진짜 목적

기능이 다 되고 나면, **일부러 아래 것들을 하나씩 해보세요.** 코드를 고치지 않아도 됩니다 — 디버거로 "지금 무슨 일이 일어나고 있는지 보는 법"을 익히는 게 목표입니다.

### 1. 두 프로세스를 동시에 디버깅하기
Visual Studio에서 서버/클라이언트 프로젝트 둘 다에 브레이크포인트를 걸어도, 기본적으로는 하나만 "시작 프로젝트"라 디버거가 안 걸릴 수 있습니다.
- 솔루션 우클릭 → 속성 → **여러 시작 프로젝트** → `EchoServer`, `EchoClient` 둘 다 "시작"으로 설정 후 F5
- 또는: 서버를 `dotnet run`으로 터미널에서 그냥 실행해두고, VS에서 **디버그 → 프로세스에 연결(Attach to Process)** 로 실행 중인 `EchoServer.exe`에 붙기

### 2. 바이트 배열을 실제로 눈으로 확인하기
`HandleClientAsync`에서 `stream.ReadAsync`가 끝난 직후 줄에 브레이크포인트를 걸고:
- **조사식(Watch) 창**에 `readBuffer` 를 추가해보세요 — 배열 그대로는 안 읽히니, `System.Text.Encoding.ASCII.GetString(readBuffer, 0, bytesRead)` 를 조사식에 직접 입력해서 "지금 들어온 바이트가 실제로 어떤 글자인지" 확인해보세요.
- STX(0x02), ETX(0x03)는 화면에 안 보이는 제어문자라서, 문자열로 보면 이상하게 보일 수 있습니다. 그럴 땐 배열을 16진수로 보는 법도 찾아보세요 (조사식에 `readBuffer` 입력 후 확장, 혹은 `BitConverter.ToString(readBuffer, 0, bytesRead)`).

### 3. 청크가 쪼개지는 순간을 직접 목격하기
5000자짜리 메시지를 보낼 때, 서버의 `ReadAsync` 직후에 브레이크포인트를 걸어두면 **한 번에 5000자가 다 안 들어오고, 여러 번에 걸쳐서 나눠 들어오는 걸** 직접 볼 수 있을 겁니다 (`bytesRead`가 매번 1024 이하로 찍힐 거예요 — 왜 1024인지도 코드에서 찾아보세요).
- 이 브레이크포인트에 **조건(Conditional Breakpoint)** 을 걸어서, "누적버퍼 안에 아직 ETX가 없을 때만" 멈추게 해보세요 (우클릭 → 조건). 이러면 완성된 메시지가 처리될 때는 안 멈추고, "아직 덜 받은" 상태에서만 멈춰서 관찰하기 편합니다.

### 4. Threads 창으로 "누가 실행 중인지" 보기
클라이언트를 2개 동시에 띄워서 서버에 접속시켜보세요 (터미널 3개: 서버 1 + 클라이언트 2). 서버 쪽 `ProcessOneMessageAsync`에 브레이크포인트를 걸고, **디버그 → 창 → 스레드(Threads)** 창을 열어보세요. 클라이언트마다 다른 스레드(혹은 async 컨텍스트)에서 처리되고 있는 걸 확인할 수 있을 겁니다 — `Task.Run`으로 클라이언트별 처리를 분리했기 때문입니다.

### 5. `await` 위에서 "한 단계씩 들어가기(Step Into, F11)" 해보기
`await stream.ReadAsync(...)` 같은 줄에서 F11(Step Into)을 눌러보세요. 일반 메서드처럼 한 줄씩 안 넘어가고 컴파일러가 만든 상태머신 코드로 들어가는 걸 볼 수 있는데, 당황하지 마시고 F10(Step Over)으로 바꿔서 "이 줄 자체가 끝날 때까지"만 넘기는 연습을 해보세요. **비동기 코드 디버깅의 기본 요령은 F11보다 F10을 훨씬 많이 쓰는 것**입니다.
